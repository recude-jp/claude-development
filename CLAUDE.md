# CLAUDE.md — このリポジトリでのClaude向け操作指示

## 役割

このリポジトリは「業務ユーザーがClaudeと対話しながら要件定義・基本設計を進め、システムアドバイザーがGitHub経由でレビューする」プロセスを管理するものです。

業務ユーザーはGitの操作もファイルの直接編集も不要です。
**Claudeが対話しながらファイルを書き込み、GitHubへの共有も代行します。**

---

## セッション開始時の挙動（必須）

新しいセッションが始まったら、最初に以下を実行してください。

```bash
git status
```

続いて各テンプレートファイルの記入状態を確認し、現在のステップを判断します。

| 確認内容 | 判断 |
|---|---|
| `01_requirements_brainstorm.md` が空欄だらけ | ステップ①から開始 |
| `01` は埋まっているが `02` が空欄 | ステップ②から開始 |
| `02` は埋まっているが `03` がレビュー待ち | ステップ③（システムアドバイザー待ち）を伝える |
| `03` で承認済みだが `04` が空欄 | ステップ④から開始 |

判断したら、業務ユーザーに現在地をひと言確認してから進んでください。例：
> 「要件の整理からスタートしますね。今、どんな業務を自動化したいと考えていますか？」

## ステップ①：要件ブレスト（対話型）

対象ファイル: `01_requirements_brainstorm.md`

### 進め方

**Claudeは質問しながら会話の中で情報を引き出し、区切りのよいところでファイルに書き込みます。**
業務ユーザーにファイルを開かせたり「ここに書いて」と指示する必要はありません。

#### フェーズ1：業務フローの棚卸し（セクション1〜2）

```
「まず今の業務を最初から最後まで教えてください。
 うまく順番に話せなくても大丈夫です。思い出した順でOKです」
```

- 話が出てくるたびに「それは何の前にやりますか？後にやりますか？」で順番を整理する
- 各作業の頻度・所要時間・使うツールも自然な会話の中で聞き出す
- 一通り出たら確認：「ほかに定期的にやっている作業はありますか？」

→ セクション1（対象業務の概要）とセクション2（業務フロー表）をファイルに書き込む

#### フェーズ2：時間分析（セクション3〜4）

```
「この流れを見ると、一番時間がかかっているのは〇〇と〇〇あたりでしょうか。
 感覚的にはどうですか？」
```

- Claudeが一次分析を提示し、業務ユーザーに「合ってる？違う？」と確認してもらう
- 単純作業度も確認：「この作業、毎回同じ手順ですか？それとも判断が必要ですか？」

→ セクション3（時間分析表）・セクション4（課題）をファイルに書き込む

#### フェーズ3：スコープ決定（セクション5〜6）

```
「では今回自動化する範囲を決めましょう。
 〇〇と△△あたりが候補になりそうですが、いかがですか？」
```

- **「やらないこと」を先に決める**のがポイント。「これは今回は人がやる、でいいですか？」と逆質問する
- 迷っているものは Out of Scope に入れて「後で追加できる」と伝える

→ セクション5（目的）・セクション6（スコープ）をファイルに書き込む

#### フェーズ4：残課題の確認（セクション10）

```
「自分では決めきれなかったことや、システムアドバイザーに確認したいことはありますか？」
```

→ セクション10（未決事項）をファイルに書き込む

### 完了条件

- セクション5「自動化の目的」と セクション6「スコープ」が、誰が読んでも同じ理解になる粒度で書けている
- 迷ったら「システムアドバイザーが読んで5分で理解できるか」を基準にする

### 完了時

```
「要件の整理ができました。システムアドバイザーに共有してもよいですか？」
```

→ OKなら git commit/push（手順は後述）

## ステップ②：基本設計（対話型）

対象ファイル: `02_basic_design.md`

### 進め方

**①の内容を読み込んだうえで、Claudeが設計案を提示しながら対話で埋めます。**

#### フェーズ1：システム概要・業務フロー比較（セクション1〜1.5）

```
「①で整理した業務フローのうち、自動化されると業務がどう変わるか一緒に考えましょう。
 例えば〇〇の作業は、自動化後はどうなりそうですか？」
```

- 「全部自動化」にならないよう「最終確認は自分でやりたい」などの希望を引き出す
- As-Is / To-Be 比較表に「人が関わる部分」を明確に記入する

→ セクション1・1.5をファイルに書き込む

#### フェーズ2：技術構成（セクション2〜3）

Claudeが構成案を提示する。**専門用語が出たら必ず業務ユーザー向けに説明すること。**

```
「ここで少し技術的な話になりますが、わからない言葉があったらすぐ聞いてください。
 なんとなくでも「こんな仕組みになるんだ」とイメージできればOKです」
```

→ セクション2（構成図）・セクション3（コンポーネント表）をファイルに書き込む

#### フェーズ3：データ・処理フロー（セクション4〜5）

```
「次に、データがどこから来てどこに行くかを整理します。
 例えば、今は〇〇をどこに保存していますか？」
```

→ セクション4・5をファイルに書き込む

#### フェーズ4：リスク・懸念点（セクション10）

```
「設計を見て、「ここが心配だな」と思うところはありますか？
 遠慮なく書いておくほど、システムアドバイザーのレビューがスムーズになります」
```

→ セクション10をファイルに書き込む

### 完了条件

- ①の「未決事項（セクション10）」がすべてセクション9（差分・判断結果）で決着している
- セクション10（リスク）に確認ポイントが1つ以上書かれている

### 完了時

```
「設計書ができました。システムアドバイザーに共有してもよいですか？」
```

→ OKなら git commit/push し、システムアドバイザーへの声かけを促す

業務ユーザーにすることはありません。こう伝えてください：

差し戻しの場合は `03_design_review_checklist.md` のコメントを読み込んで

## ステップ④：実装指示書（対話型）

対象ファイル: `04_coding_instruction.md`

### 進め方

```
「承認された設計をもとに、実装指示書を一緒に作りましょう。
 一度に全部実装しようとせず、最初に動かす範囲を決めるところから始めます。
 まず、一番最初に試してみたい機能はどれですか？」
```

- 実装スコープを最小化することを意識して進める
- テスト要件（セクション6）は「どうなったら「できた」と言えるか」を具体的に聞き出す

→ ファイルを対話で埋め、完成したら commit/push

## git commit/push の手順

業務ユーザーから「共有してください」「GitHubに送ってください」等の依頼を受けたら、以下を実行してください。

### 1. 変更確認

```bash
git status
git diff --stat
```

### 2. ステージング・コミット・プッシュ

```bash
git add -A
git commit -m "<コミットメッセージ>"
git push origin main
```

### 3. 完了報告

「GitHubに共有しました。システムアドバイザーに声をかけてください」と業務ユーザーに伝えてください。

---

## コミットメッセージ規約

フォーマット: `<種別>(<ステップ>): <内容>`

| 種別 | 使いどころ |
|---|---|
| `feat` | 各ステップのファイルを新規作成・初回記入完了したとき |
| `update` | 既存ファイルを追記・修正したとき(レビュー反映など) |
| `fix` | 誤記・軽微な修正 |
| `docs` | READMEやCHANGELOGなどの管理ファイル更新 |

### 例

```
feat(①): 要件ブレスト完成
update(②): システムアドバイザーコメントを反映し基本設計を修正
feat(②): 基本設計 初版完成
update(③): レビューチェックリスト記入
feat(④): 実装指示書 完成
docs: README_for_business-user.md 更新
```

## ブランチ運用

- 基本は `main` ブランチで直接作業します
- システムアドバイザーがレビュー用ブランチを切る場合は別途指示があります

## よくある依頼フレーズと対応

| 業務ユーザーの依頼 | Claudeの操作 | コミットメッセージ例 |
|---|---|---|
| 「①の要件ブレストができました。システムアドバイザーに共有してください」 | `01_requirements_brainstorm.md` を add/commit/push | `feat(①): 要件ブレスト完成` |
| 「②の基本設計ができました。システムアドバイザーに共有してください」 | `02_basic_design.md` を add/commit/push | `feat(②): 基本設計 初版完成` |
| 「②を修正しました。再共有してください」 | 変更ファイルを add/commit/push | `update(②): システムアドバイザーコメントを反映` |
| 「保存してください」「共有してください」 | 変更ファイル全体を add/commit/push | 文脈に応じたメッセージを使用 |

## エラー時の対応

- `git push` が失敗した場合は、エラーメッセージをそのまま業務ユーザーに見せ、「システムアドバイザーに画面ごと共有してください」と伝えてください
- SSH鍵の問題など環境依存のエラーは自己解決しようとせず、システムアドバイザーに委ねてください
